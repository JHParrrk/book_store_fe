# 🔄 핵심 동작 원리 및 데이터 흐름 가이드 (Core Flow Guide)

이 문서는 애플리케이션의 뼈대를 이루는 3가지 핵심 기술 흐름이 프로젝트 내에서 어떻게 동작하고 맞물려 있는지, **[프로젝트 라이브러리 총정리 가이드]**의 핵심 패키지들과 연계하여 아주 상세하게 설명합니다.

---

## 1. 🚦 라우팅 및 화면 전환 흐름 (React Router DOM)

React는 기본적으로 하나의 HTML 파일(`index.html`)만 사용하는 SPA(Single Page Application)입니다. URL이 바뀔 때 HTML을 새로 받아오는 것이 아니라, **`react-router-dom`** 패키지가 URL을 가로채서 그에 맞는 자바스크립트 컴포넌트(페이지)만 부드럽게 교체해 줍니다. 깜빡임 없는 극상의 UX를 제공하는 핵심입니다.

### 동작 원리 및 세부 흐름
1. **라우터 정의 (`src/router.tsx`)**:
   - `createBrowserRouter`를 사용하여 브라우저의 URL 경로(`path`)와 화면에 그릴 컴포넌트(`element`)를 연결(매핑)합니다.
2. **동적 파라미터 매핑**:
   - `/books/:bookId` 와 같이 동적 주소를 설정하여, 1번 책, 2번 책에 따라 하나의 상세 페이지 컴포넌트(`<BookDetail />`)가 데이터만 갈아끼운 채 재사용되도록 설계되어 있습니다.
3. **공통 레이아웃 (Nested Routing 흐름)**:
   - 모든 페이지는 단순히 렌더링되지 않고 `<Layout>` 컴포넌트로 감싸집니다.
   - 브라우저 URL이 변경되면 Layout의 뼈대(Header, Footer)는 굳건히 유지된 채, 중간 비어있는 공간(Children 구역)에만 알맞은 컴포넌트(`<Home>`, `<Login>` 등)가 교체되며 렌더링됩니다.
4. **앱에 주입 (`src/App.tsx`)**:
   - `<RouterProvider router={router} />`를 통해 이 라우팅 규칙들을 앱 전역에 주입하여, 앱의 네비게이션 시스템을 가동합니다.

## 2. 🎨 전역 스타일링 및 테마 아키텍처 (CSS + Theme Context 흐름)

단순한 다크 모드 스위치를 넘어, 브라우저 기본 스타일 초기화부터 컴포넌트단까지 **`@vanilla-extract/css`**, **`Theme Context`**, **`sanitize.css`**가 어떻게 폭포수처럼 아래로 흘러가며 디자인을 완성하는지 보여주는 전체 흐름입니다.

### CSS 렌더링 및 테마 적용의 폭포수(Cascade) 4단계 흐름
1. **브라우저 초기화 (`sanitize.css`)**:
   - 프로젝트 진입점(`index.tsx`)에서 가장 먼저 로드되어, 크롬/사파리/엣지 등 브라우저마다 제각각인 기본 여백 및 폰트 크기 등의 "찌꺼기"를 깨끗하게 밀어버립니다(바닐라 상태로 평준화).
2. **전역 스타일 및 테마 계약 (Vanilla Extract: `theme.css.ts`, `global.ts`)**:
   - `createThemeContract` 기능으로 애플리케이션에서 쓸 색상들의 '껍데기(인터페이스)'를 단단하게 규정합니다. (예: `primary`, `background`)
   - 라이트 모드용 팔레트와 다크 모드용 팔레트를 각각 선언하여, 이를 빌드 시점에 완벽한 "순수 CSS 변수"로 변환해 놓습니다.
3. **테마 상태 제어 및 스위치 (`Theme Context`)**:
   - 사용자가 첫 접속 시 브라우저 내부(`localStorage`)를 뒤져 이전 테마 설정값을 기억해 냅니다.
   - 사용자가 헤더에서 ☀️/🌙 (ThemeSwitcher) 버튼을 누르면 이 Context의 전역 상태(`themeName`)가 바뀝니다.
   - 상태가 바뀌면, `App.tsx`의 `<BookStoreThemeProvider>`가 최상위 전역 컨테이너에 **라이트/다크에 맞는 Vanilla Extract CSS 클래스명을 붙였다 뗐다**(토글) 합니다.
4. **개별 컴포넌트 침투 (`*.css.ts`)**:
   - 최상단에서 테마 클래스가 교체되면, 하위에 있는 수많은 컴포넌트(버튼, 입력창 등)는 내부적으로 CSS 변수(`var(--color-primary)`)로 의존하고 있기 때문에 동시에 색상이 쫙 바뀝니다.
   - 개별 컴포넌트는 자기가 다크 모드인지 라이트 모드인지 알 필요 없이 "나는 primary 색상 쓸래"라고만 정의하면 되어 **비즈니스 로직과 UI 로직이 완벽히 분리**됩니다! (이것이 Zero-runtime CSS의 최대 장점입니다)

## 3. 📡 서버 데이터 통신 및 렌더링 파이프라인 (React Query + Axios)

게시판, 책 상세 정보 등 **서버에서 가져오는 데이터(Server State)**의 흐름입니다. 단순 HTTP 통신용 라이브러리인 **`axios`**와 서버 상태 관리 끝판왕인 **`@tanstack/react-query`**가 어떻게 완벽히 역할을 분담하여 돌아가는지 보여줍니다.

### 동작 원리 및 세부 파이프라인
1. **HTTP 통신 계층 (`Axios` - `apis/https.ts` 등)**:
   - 브라우저와 백엔드 서버 간의 실제 대화를 담당하는 행동대장입니다. 기본 `fetch` 대신 `axios`를 써서 타임아웃, 공통 헤더 세팅(헤더에 항상 JWT 토큰 싣기), API 에러 가로채기(Interceptor)를 아주 똑똑하게 한 곳에서 처리합니다.
2. **서버 상태 중앙 관리소 창고 (`React Query` - `queryClient.ts`)**:
   - `queryClient` 전역 객체는 Axios가 가져온 데이터를 언제까지 최신(Fresh)으로 인정할지, 지나면 낡은(Stale) 상태로 간주할지 '캐싱 정책'을 결정합니다. "만약 에러 나면 3번까지는 다시 재시도 해라(retry)" 같은 강력한 룰을 여기서 세팅합니다.
3. **데이터 흐름 파이프라인 예시 (컴포넌트단 호출)**:
   - 개발자가 `<Books />` 컴포넌트 안에서 `useQuery` 혹은 커스텀 훅(`useBooks`)을 부릅니다.
   - ➡️ **(1) 캐시 확인**: React Query가 가장 먼저 자체 메모리(창고)를 뒤져 "혹시 방금 받아온 책 목록 데이터 있어?" 라고 확인합니다.
   - ➡️ **(2) Axios 호출**: 데이터가 한 번도 불린 적 없거나 너무 오래되어 낡았다면, Axios에게 "백엔드 뛰어가서 새로 가져와!"라고 지시합니다. 이때 컴포넌트에는 `isLoading: true`를 던져주어 알아서 화면에 `<Loading />` 뺑글이가 돌게 합니다.
   - ➡️ **(3) 상태 자동 분기 및 저장**: 인터넷이 끊기거나 서버가 죽으면 즉시 `isError: true`로 바뀌어 `<Error />` 폴백 페이지를 띄웁니다. 데이터를 정상적으로 물고 오면 화면을 그림과 동시에 **몰래 자신의 캐시 창고에 데이터를 백업**합니다.
   - ➡️ **(4) 극강의 UX (Zero-Loading)**: 사용자가 장바구니 탭에 갔다가 곧바로 다시 책 목록 탭으로 돌아오면, 서버에 재요청하지 않고 0.1초 만에 창고의 데이터를 꺼내 그려주어 속도가 모바일 네이티브 앱처럼 아주 빠릅니다.

## 🎯 완벽 합체 아키텍처 (App.tsx 프로바이더 샌드위치)

위 3가지 강력한 전역 상태 망(Provider)들이 프로젝트 최상단(`App.tsx`)에서 어떻게 애플리케이션을 보호하며 겹겹이 덮고 있는지 요약합니다. 이 순서는 데이터 흐름의 계층 구조를 띠고 있습니다.

```tsx
// 1층. 데이터 우산: 내 밑의 모든 구역은 서버 데이터와 캐싱 기능을 마음껏 빼다 쓸 수 있어! (Axios 통신 + React Query 상태 저장)
<QueryClientProvider client={queryClient}>      
  
  // 2층. 테마 우산: 내 밑의 모든 구역은 Vanilla Extract로 정의된 다크/라이트 색상을 화면 깜빡임 없이 상속받게 해줄게! (Theme Context)
  <BookStoreThemeProvider>                      
    
    // 3층. 네비게이션 우산: 위 모든 기능을 다 물려받은 상태에서, URL 경로에 맞게 홈/상세/로그인 화면으로 안전하게 진입시켜 줄게! (React Router DOM)
    <RouterProvider router={router} />          

    // 4층. 프레젠테이션 UI: 나는 페이지 전환에 상관없이 최상위에서 대기하다가, 화면 위를 떠다니며 에러/성공 토스트 알림을 띄워줄게!
    <ToastContainer />                          

  </BookStoreThemeProvider>

</QueryClientProvider>
```

이처럼 잘 짜인 **Provider 패턴 샌드위치 구조**를 통해, 개발자는 실제 화면(Pages)이나 컴포넌트를 코딩할 때 귀찮은 데이터 파이프라인, 테마 클래스 교체, 주소 변경 설정들을 매번 신경 쓰지 않고 오직 **'비즈니스 핵심 로직(화면 UI의 동작)'**에만 집중할 수 있게 됩니다. 이것이 곧 프론트엔드 엔터프라이즈급 아키텍처의 정수입니다.
